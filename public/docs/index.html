Cenvero Fleet Docs

One controller. Two transport modes. Full control — no cloud required.

Controller binary

The fleet binary manages servers, services, logs, alerts, keys, and updates from your local machine.

Agent binary

The fleet-agent binary runs on each managed server. It can be installed automatically or manually.

Direct mode

The controller SSHes into the agent on demand. Use when the controller can reach the server.

Reverse mode

The agent dials out to the controller. Use when the server is behind NAT or a firewall.

Install

The one-liner installer detects your platform and installs the fleet controller binary.

Linux & macOS

curl -fsSL https://fleet.cenvero.org/install | sh

Homebrew (macOS / Linux)

brew install cenvero/fleet/cenvero-fleet

Windows (PowerShell)

irm https://fleet.cenvero.org/install.ps1 | iex

Build from source

git clone https://github.com/cenvero/fleet
cd fleet
make build

Requires Go 1.24+. Produces fleet and fleet-agent in the repo root.

Release verification — all release archives are signed with minisign. The installer verifies the signature against the embedded public key before swapping the binary.

Initialize

Run once on the machine that will act as the controller:

fleet init

This creates the config directory (default ~/.cenvero-fleet/), generates controller keys, initialises the SQLite databases, and writes config.toml. All other fleet commands are blocked until fleet init has been run.

Verify everything looks right:

fleet status
fleet config show

Add a Server

Run fleet server add with no arguments to be guided through a step-by-step prompt, or pass the name and address directly:

# Interactive mode — prompts for all details
fleet server add

# Non-interactive — name and address provided
fleet server add web-01 192.0.2.10

# Reverse mode (agent dials out to the controller)
fleet server add edge-01 unknown --mode reverse

In interactive mode the CLI asks for name, address, transport mode, login credentials (key or password), SSH port, and sudo preference — pressing Enter accepts each default.

Auto-install Agent

For direct-mode servers the agent is automatically installed by default — no extra flags needed. When you run fleet server add (interactive or not), the CLI shows a preview of what will be installed and asks for confirmation before SSHing in.

# Fully interactive — prompts for everything including login creds
fleet server add

# Non-interactive with key auth
fleet server add web-01 192.0.2.10 \
  --login-user root \
  --login-key ~/.ssh/id_ed25519

# Non-interactive with password auth
fleet server add web-01 192.0.2.10 \
  --login-user ubuntu \
  --login-password mypassword \
  --sudo

# Skip auto-install (register only)
fleet server add web-01 192.0.2.10 --no-agent

The auto-install flow:

  1. Shows an install preview (files and services that will be created)
  2. SSHes to the server using the provided login credentials (key or password)
  3. Detects the CPU architecture (uname -m)
  4. Downloads the correct fleet-agent release binary from GitHub
  5. Installs it to /usr/local/bin/fleet-agent
  6. Creates and enables a fleet-agent.service systemd unit
  7. Starts the service and registers the server in the controller
Linux only — auto-install requires systemd. For macOS or Windows agents, pass --no-agent and follow the manual setup instructions printed by fleet server add.

Teardown

Removing a managed server automatically tears down the agent:

fleet server remove web-01

This stops and disables the fleet-agent.service, removes the binary, and deletes the server record from the controller.

Transport Modes

Direct mode

Controller initiates an SSH connection to the agent on each operation. The server must be reachable from the controller's network. Best for servers in the same network or with public IPs.

Reverse mode

The agent dials out to the controller's reverse listener and holds the connection open. The controller sends RPCs through the established tunnel. Best for NAT'd or firewalled servers.

You can mix both modes in one deployment. Start the controller daemon for reverse sessions:

fleet daemon

Start a reverse agent on the remote server:

fleet-agent reverse \
  --controller controller.example.com:9443 \
  --server-name edge-01

Shell Access

Open an interactive root shell on any managed server:

fleet ssh web-01

This invokes the system ssh binary using the fleet controller key and connects to the agent's SSH port (default 2222) with the fleet host key pinned. No separate SSH key setup or host verification needed.

Root access — the agent runs as root and fleet always connects as root. This is required for systemd, firewall, and privileged operations.

Run Commands

Run a one-off shell command on a single server:

fleet exec web-01 uptime
fleet exec web-01 "df -h /"
fleet exec web-01 "cat /etc/os-release"
fleet exec web-01 "systemctl status nginx"

Run the same command across all servers concurrently:

fleet exec --all uptime
fleet exec --all "free -m"
fleet exec --all "apt list --upgradable 2>/dev/null | head"

Output is shown per-server with the server name prefix. Non-zero exit codes are reported as errors. Commands run through /bin/sh -c on the agent.

Services

Track and control systemd services on Linux agents. linux

Track a service

# Add with log file tracking
fleet service add web-01 nginx.service \
  --log /var/log/nginx/access.log \
  --critical

# List tracked services
fleet service list web-01

Control services

fleet service start   web-01 nginx.service
fleet service stop    web-01 nginx.service
fleet service restart web-01 nginx.service

Logs

Read tracked service logs live or from the controller-side cache.

Live log stream

# Read recent lines
fleet service logs web-01 nginx.service

# Follow in real time
fleet service logs web-01 nginx.service --follow

Cached logs

The controller caches log segments locally. Access them even when the server is offline:

fleet service logs web-01 nginx.service --cached

The cache is retained by size, count, and age. Defaults can be changed in config.toml.

Firewall & Ports

Manage UFW rules and open/close ports on Linux agents. linux

Firewall

fleet firewall status  web-01
fleet firewall enable  web-01
fleet firewall disable web-01
fleet firewall add     web-01 "allow 443/tcp"
fleet firewall remove  web-01 "allow 443/tcp"

Ports

fleet port list  web-01
fleet port open  web-01 443
fleet port close web-01 443

Alerts

Cenvero Fleet evaluates metric thresholds and fires alerts when servers breach them.

View and manage alerts

# List all active alerts
fleet alerts

# Filter by server
fleet alerts --server web-01

# Acknowledge an alert
fleet alerts ack <id>

# Suppress for a duration
fleet alerts suppress <id> --for 6h

Dashboard

The fleet dashboard command opens a full-terminal TUI with multiple panels:

fleet dashboard
PanelDescription
OverviewController status, server count, alert summary
ServersPer-server health, mode, last seen, metrics
ServicesService status across all tracked servers
LogsRecent log lines, cached and live
AlertsActive alerts with ack and suppress actions
OpsQuick access to common operations

Keyboard navigation with arrow keys and Tab. Mouse click supported. Press q or Ctrl+C to exit.

Keys

The controller uses Ed25519 keys for SSH authentication. Inspect and rotate them:

# Show controller key fingerprint
fleet key fingerprint

# Rotate the controller key
fleet key rotate

Key rotation pushes the new public key to all connected servers' authorized_keys before replacing the local private key. Rollout is safe for both direct and reverse fleets.

Updates

Cenvero Fleet manages its own updates. Agents are automatically kept in sync with the controller version whenever they connect.

# Check for available updates
fleet update check

# Apply the latest release
fleet update apply

# Roll back to the previous version
fleet update rollback

# Switch update channel
fleet update channel stable
fleet update channel beta

Available channels: stable, beta, alpha. Update artifacts are signature-verified before applying.

Homebrew installs — if fleet was installed via Homebrew, fleet update apply will skip self-update and show the correct brew upgrade cenvero-fleet command instead. Agents are still updated regardless.

Configuration

# Show current config
fleet config show

# Validate config file syntax
fleet config validate

# Backup and restore
fleet config backup
fleet config restore

# Export / import (for migration)
fleet config export
fleet config import

Database backend

The default backend is SQLite. To shift to PostgreSQL, MySQL, or MariaDB:

fleet database show
fleet database shift \
  --backend postgres \
  --dsn 'postgres://user:pass@host:5432/fleet?sslmode=require'

Directory Layout

Default location: ~/.cenvero-fleet/

~/.cenvero-fleet/
├── config.toml          # controller configuration
├── instance.id          # stable instance identifier
├── keys/
│   ├── id_ed25519       # controller private key
│   ├── id_ed25519.pub
│   ├── known_hosts      # pinned agent host keys
│   └── rotations/       # previous key versions
├── servers/             # per-server TOML records
├── templates/           # reusable server templates
├── logs/
│   ├── _aggregated/     # cached remote log segments
│   └── _audit.log       # controller audit log
├── alerts/              # alert state store
├── data/
│   ├── state.db         # server and service state
│   ├── metrics.db       # metrics history
│   └── events.db        # event log
├── backups/             # config backup archives
└── tmp/
